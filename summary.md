Root cause: native fork pre-creation in [run_cmd_fork.rs](/home/obj/project/github/RyderFreeman4Logos/cli-sub-agent/crates/cli-sub-agent/src/run_cmd_fork.rs:298) was using `csa_session::create_session()`, which is daemon-aware and reuses `CSA_DAEMON_SESSION_ID` when present. In daemon-backed `csa run` flows, that let a fork child accidentally reuse the parent ULID; later `cleanup_pre_created_fork_session()` in the same file could delete the parent session directory outright, which matches the observed symptom: local branch/commits preserved, but `sessions/<id>/` vanished and `session wait` started returning “not found”.

Fix: switch native fork pre-creation to `csa_session::create_session_fresh()` and add a regression test that simulates daemon env leakage, proves the child gets a distinct ULID, and verifies child cleanup no longer deletes the parent session.
